# 剧本 YAML Schema 设计说明

## 设计原则
本 Schema 旨在平衡**可读性**、**可扩展性**和**编剧行业习惯**，使小说改编的剧本既便于机器处理，又符合人工编辑习惯。

---

## 结构设计原因

### 1. 三层结构（meta → characters → scenes）
**设计原因**：
- **meta**：独立元信息层，便于剧本管理和版本控制
- **characters**：集中人物列表，避免在每个场景重复定义，便于统一修改人物设定
- **scenes**：按场景组织，符合剧本创作的线性叙事习惯

### 2. 场景内元素使用 oneOf 多态
**设计原因**：
- 支持四种核心元素类型：`dialogue`（对话）、`action`（动作）、`narration`（旁白）、`transition`（转场）
- 通过 `type` 字段明确区分，易于扩展新类型（如音效、镜头等）
- 保持元素列表的线性顺序，符合剧本阅读习惯

### 3. 对话元素包含 tone 和 action 字段
**设计原因**：
- `tone`：表达情感音调，是小说改编剧本的关键（小说中的心理描写常转化为语气）
- `action`：伴随动作，使对话更生动，符合影视剧本规范

### 4. 动作元素的 character 可选
**设计原因**：
- 有人物的动作：如"张三推开门"
- 纯场景动作：如"雷声大作"、"灯光亮起"

---

## 字段选择依据

| 字段 | 设计依据 |
|------|---------|
| `meta.title` | 剧本必备信息 |
| `meta.author/adaptedBy` | 版权信息 |
| `characters[].description` | 便于保持人物一致性 |
| `scenes[].time/location` | 剧本标准三要素（时间、地点、人物） |
| `scenes[].id` | 便于引用和跳转 |

---

## 扩展性考虑
- 预留新元素类型入口（只需在 oneOf 中添加）
- 所有描述字段均为 string，支持富文本（如 Markdown）
- 可选字段多，便于按需填充
