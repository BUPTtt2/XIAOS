# 方案选择

## 方案对比

### 方案 A：Node.js 后端 + 纯前端
**大致怎么做**：
- 前端：HTML/CSS/JS（使用 IndexedDB 本地存储）
- 后端：Express.js + Node.js（代理 ModelScope API）
- 通信：REST API
- 存储：前端 IndexedDB + 后端临时缓存

**主要优点**：
- 技术栈简单，快速开发
- 本地运行无依赖
- API Key 安全隐藏在后端

**主要风险**：
- 需要同时维护前后端
- 本地数据持久化需要 IndexedDB 方案

**实现复杂度**：Medium
**可测试性**：好

---

### 方案 B：全栈 Next.js（推荐）
**大致怎么做**：
- Next.js 15（App Router），完整全栈框架
- 前端：React + Tailwind CSS
- 后端：Next.js API Routes（代理 ModelScope）
- 存储：SQLite + Prisma（本地持久化）
- 用户认证：NextAuth.js（简单本地认证）

**主要优点**：
- 单一代码库，部署简单
- 本地 SQLite 稳定持久化
- 内置路由、API 路由、组件化
- 生态完善，组件丰富

**主要风险**：
- 技术栈稍重，但本地运行无问题
- 学习曲线略高（但文档完善）

**实现复杂度**：Medium
**可测试性**：好

---

## 最终选择：方案 B（Next.js）

**选择理由**：
1. 单一代码库，便于 PR 拆分（前后端在一个仓库）
2. 本地 SQLite 持久化，历史记录稳定
3. Next.js API Routes 完美适配代理 ModelScope
4. 组件化开发，便于可视化编辑器实现
5. 生态完善，快速迭代

## PR 拆分规划

| PR | 内容 |
|----|------|
| PR1 | 项目初始化 + YAML Schema 定义 |
| PR2 | 前端基础框架 + 文件上传 |
| PR3 | AI 转换模块（ModelScope 集成） |
| PR4 | 可视化编辑器 + YAML 预览 |
| PR5 | 用户系统 + API Key 管理 |
| PR6 | 历史记录功能 |
| PR7 | 角色管理 + 优化完善 |
